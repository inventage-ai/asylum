## Context

The config TUI (`asylum config`) lets users toggle kits on/off. Currently, activation removes a YAML comment block and inserts the kit's `ConfigSnippet`; deactivation removes the YAML entry and inserts a comment. This is broken for opt-in kits because their `ConfigSnippet` is commented-out (it represents the default state for new configs), so activation re-inserts a comment instead of an active entry.

The `KitConfig` struct already has a `Disabled *bool` field, and `KitActive()`/`DisabledKits()`/`kit.Resolve()` already respect it. The field just isn't wired into the config TUI.

## Goals / Non-Goals

**Goals:**
- Fix kit activation for opt-in kits in the config TUI
- Use `disabled: true` for toggling previously-enabled kits instead of comment manipulation
- Preserve comment blocks for never-enabled kits (hand-editable config)
- Enable project-level override of globally-disabled kits via `disabled: false`

**Non-Goals:**
- Changing the `disabled` field name (it's already in use and the semantics are clean)
- Auto-populating all kits into the config (discovery stays in the TUI)
- Changing how `KitActive`/`DisabledKits`/`kit.Resolve` work (they're already correct)

## Decisions

### Kit state representation in config file

Three states, determined by file content:

| State | Config file representation | `KitActive()` |
|-------|---------------------------|----------------|
| Never enabled | Comment block (`# ast-grep:`) | false |
| Active | YAML entry, no `disabled` field | true |
| Disabled | YAML entry with `disabled: true` | false |

Absent `disabled` field = active (backward compatible). This means existing configs and project-level `.asylum` files work without changes.

**Alternative considered**: Adding `enabled: true/false` instead. Rejected because `disabled` already exists in the struct and the semantics are cleaner — the exceptional state (disabled) gets the annotation, the normal state (active) is clean with no field.

### Config TUI toggle logic

The TUI needs to distinguish "never enabled" from "disabled" to choose the right persistence strategy:

```
Toggle ON:
  if kit has comment block → remove comment, insert "  <name>:"    (first activation)
  if kit has entry with disabled:true → remove disabled field       (re-enable)

Toggle OFF:
  if kit has comment block → no-op (already off)
  if kit has active entry → add "disabled: true" as first property  (disable)
```

After the first activation, the kit entry persists forever — only the `disabled` field toggles. The comment block is a one-way door: once removed, it doesn't come back.

### Detecting kit state in the config file

The config TUI currently builds `activeKits` from `cfg.KitNames()` (all keys in the merged `Kits` map). This is wrong after this change because disabled kits are also in the map.

Fix: use `cfg.KitActive(name)` instead of map membership to determine pre-selection state.

Additionally, the TUI needs to know whether a kit has ever been enabled in the **global** config (has a YAML entry, even if disabled) vs never enabled (only exists as a comment). This determines whether toggling ON should remove a comment or flip `disabled`.

Approach: check if the kit key exists in the global config file's `kits` mapping (via `parseConfigDoc` + `kitExistsInMapping`). If it exists → flip disabled. If not → remove comment + insert entry.

### Text-based disabled field manipulation

New functions in `internal/config/sync.go` (or `isolation.go`, same pattern):

- `SetKitDisabled(path, kitName string) error` — adds `disabled: true` as the first property under the kit's entry. Uses the existing `findSection`/`findKey`/`insertAfter` helpers from `isolation.go`.
- `RemoveKitDisabled(path, kitName string) error` — finds and removes the `disabled: true` line from the kit's entry.

These follow the same text-based editing pattern as `SetKitCredentials` and `SetAgentIsolation`.

### Inserting disabled as first property

When disabling, `disabled: true` should be the first property under the kit key for visibility:

```yaml
kits:
  apt:
    disabled: true          # immediately visible
    packages:
      - imagemagick
```

The `insertAfter(lines, kitIdx, indent, "disabled: true")` pattern naturally inserts right after the kit key line, which is the first position.

## Risks / Trade-offs

- **[Risk] Comment format variations** — Users who hand-edit comments with different spacing could cause `RemoveKitComment` to miss the block. → Mitigation: This is an existing risk, unchanged by this design. The comment format is generated by asylum, so it's consistent unless manually edited.
- **[Trade-off] Comment blocks are one-way** — Once a kit is activated, the comment block (with example config) is gone forever. Re-disabling uses `disabled: true` instead. → Acceptable: the comment was a template, not user config. The TUI is the discovery mechanism.
- **[Trade-off] Mixed state in config file** — Some inactive kits are comments, others have `disabled: true`. → Acceptable: this reflects the real lifecycle (never-enabled vs previously-enabled) and is more informative than a uniform representation.
